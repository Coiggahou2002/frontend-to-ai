# GPU 与大模型选型指南

## 为什么大模型需要 GPU

CPU 擅长复杂逻辑和分支判断（少量核心，每个核心很强），GPU 擅长大规模并行计算（数千个小核心同时运算）。大模型的本质是海量矩阵乘法——正好是 GPU 的强项。

类比：CPU 像一个顶级厨师（做什么菜都行，但一次只能炒一盘），GPU 像 5000 个学徒同时切菜（每个人只会简单操作，但并行起来速度极快）。

---

## GPU 最核心的指标：显存（VRAM）

### 显存是什么

GPU 显存就是 GPU 自己的内存。模型的所有参数（权重）必须加载到显存中才能运行。**显存大小决定了能跑多大的模型**，这是选型时最重要的一个指标。

注意区分：
- **系统内存（RAM）**：服务器上的 CPU 内存，比如 30 GiB / 60 GiB / 128 GiB
- **GPU 显存（VRAM）**：需要根据 GPU 型号查，比如 A10 = 24 GB，A100 = 80 GB

### 模型参数量与显存的关系

大模型的"7B""32B"中的 B = Billion（十亿），指模型有多少个参数。每个参数需要占用一定的存储空间，占多少取决于**精度（数据类型）**：

| 精度 | 每个参数占用 | 含义 | 7B 模型显存 | 32B 模型显存 |
|------|------------|------|-----------|-------------|
| FP32 | 4 字节 | 32位浮点数（训练用，推理很少用） | 28 GB | 128 GB |
| FP16 / BF16 | 2 字节 | 16位浮点数（标准推理精度） | **14 GB** | **64 GB** |
| FP8 | 1 字节 | 8位浮点数（质量接近FP16，显存减半） | 7 GB | **32 GB** |
| INT8 | 1 字节 | 8位整数量化 | 7 GB | 32 GB |
| INT4 | 0.5 字节 | 4位整数量化（质量有损） | 3.5 GB | 16 GB |

**简单公式：显存需求 ≈ 参数量 x 每参数字节数 + 额外开销**

额外开销包括：
- **KV Cache**：模型在对话中需要记住上下文，这部分也占显存，上下文越长占越多
- **推理框架本身**：vLLM、TGI 等框架也需要一些显存
- **操作系统预留**：通常 GPU 会预留 0.5-1 GB

经验法则：**实际所需显存 ≈ 模型权重大小 x 1.2**（预留 20% 余量给 KV Cache 等）。

### 精度越低越好吗？

不是。降低精度（量化）本质上是**用质量换显存和速度**：

| 精度 | 质量损耗 | 适用场景 |
|------|---------|---------|
| FP16 / BF16 | 无损 | 预算充足时的首选 |
| FP8 | 极微（通常感知不到） | **性价比最优，推荐** |
| INT8 | 轻微 | 显存紧张时的实用选择 |
| INT4 | 明显（复杂推理、细微语义会退化） | 仅在显存实在不够时使用 |

对于大部分生产场景（实体提取、文本生成、摘要等），FP8 已经完全够用。

## 多卡部署：什么时候需要多张 GPU

当一张 GPU 的显存装不下整个模型时，需要把模型拆分到多张 GPU 上。这叫 **Tensor Parallelism（张量并行）**。

| 情况 | 方案 |
|------|------|
| 模型 < 单卡显存 | 1 张卡，最简单 |
| 模型 > 单卡显存 | 多张卡做张量并行，模型切片分布到各卡上 |

举例：
- Qwen3-32B FP16 需要 ~64 GB → 1x A100 80GB（够）或 2x A100 40GB（拆分）
- Qwen3-235B FP8 需要 ~235 GB → 至少 4x A100 80GB（320 GB）

**多卡的代价**：
1. 卡间通信有开销，速度不是线性提升
2. 价格通常是线性甚至超线性增长（4卡实例不止4倍价格）
3. 配置和调试更复杂

**能用单卡解决的问题，不要用多卡。**

## MoE 模型的特殊性

你可能听说过 Qwen3-235B-A22B 这个模型，它的架构叫 **MoE（Mixture of Experts，混合专家）**。

普通模型（Dense）：每次推理时，所有参数都参与计算。
MoE 模型：模型分成很多个"专家"，每次推理只激活其中一小部分。

Qwen3-235B-A22B 的含义：
- **235B**：总参数量 2350 亿
- **A22B**：每次推理只激活 220 亿参数

听起来很高效，但有个陷阱：**所有 235B 的参数都必须常驻显存**，即使每次只用 22B。所以它对显存的需求是按 235B 算的，不是 22B。

| 指标 | Qwen3-32B (Dense) | Qwen3-235B-A22B (MoE) |
|------|-------------------|----------------------|
| 推理速度感受 | 类似 32B 的速度 | 类似 22B 的速度（更快） |
| 显存需求 | 按 32B 算 | **按 235B 算**（大得多） |
| 输出质量 | 很好 | 更好（但对大部分任务差距不大） |
| 部署成本 | 中等 | 非常高（成本约为 Dense 的 4 倍以上） |

## 除了显存，还需要关注什么

### 系统内存（RAM）

系统内存需要 ≥ GPU 显存，因为模型加载时会先读入系统内存再传到 GPU。如果系统内存不够，模型加载会失败。

一般云平台的 GPU 实例都已经搭配了足够的系统内存，不需要额外操心。

### vCPU

CPU 在模型推理中的角色主要是：数据预处理（tokenization）、网络 IO、调度。对推理速度影响不大。通常 **8-16 vCPU** 配 1 张 GPU 就足够了。

### 磁盘

模型文件需要存储在磁盘上，大模型动辄几十 GB。例如：
- 8B 模型：~16 GB 模型文件
- 32B 模型：~64 GB 模型文件（FP16 权重）

云服务器默认的系统盘通常是 40-100 GB，可能不够放大模型。**建议额外挂载 200 GB 以上的 SSD 数据盘**。

### 模型尺寸与能力分档

| 参数量 | 能力水平 | 擅长的任务 | 吃力的任务 |
|-------|---------|-----------|-----------|
| 1.5B-4B | 初级 | 简单分类、关键词提取、短文本摘要 | 复杂推理、长文写作、风格模仿 |
| **7B-8B** | 中等 | 实体提取、基础问答、模板化生成 | 细腻的语气把控、多轮复杂推理 |
| **14B** | 中上 | 大部分 NLP 任务都能胜任 | 极复杂的长链推理 |
| **32B** | 强 | 复杂写作、风格模仿、多要素综合推理 | 几乎没有短板 |
| 70B+ | 很强 | 全面能力接近闭源大模型 | 成本和部署门槛极高 |

模型越大，理解力和生成质量越好，但成本也越高。选型的关键是**找到"刚好够用"的最小尺寸**。

### 选型原则：由最难的任务决定

一个系统中通常有多个 LLM 任务，模型尺寸应该由**最难的那个任务**决定。简单任务用大模型跑没问题（只是浪费），但难任务用小模型跑会出问题（质量不达标）。

以一个典型的 AI 应用为例，假设系统需要做实体提取、内容摘要、风格改写和长文生成：

| 任务 | 复杂度 | 最低够用 | 推荐 | 为什么 |
|------|-------|---------|------|-------|
| 实体提取（人名、地点等） | 低-中 | 7B | 7B-14B | 结构化输出，模式固定 |
| 内容摘要 | 低-中 | 7B | 7B-14B | 提取关键信息，模式相对固定 |
| 风格改写（模仿特定语气） | 中-高 | 14B | 14B-32B | 需要理解风格的细微差异 |
| **长文生成**（综合多种上下文） | **最高** | 14B | **32B** | 要结合多种信息源，保持连贯性和风格一致性 |

瓶颈在长文生成——它要把多种信息揉在一起、用特定风格生成自然的输出，对模型综合能力要求最高。

### 尺寸与成本的跳变

模型尺寸不是线性影响成本，而是存在**GPU 档位跳变**：

| 模型尺寸 | 最低 GPU | 关键跳变 |
|---------|---------|---------|
| 7B-8B | 1x A10 (24GB) | -- |
| 14B (INT8) | 1x A10 (24GB) | 还能挤进 A10 |
| 14B (FP8) | 1x A100 (80GB) | **跳到 A100，成本约为 A10 的 5-6 倍** |
| 32B (FP8) | 1x A100 (80GB) | 同 A100，和 14B FP8 成本一样 |
| 70B (FP8) | 1x A100 (80GB) | 勉强能装，余量很小 |
| 235B MoE (FP8) | 4x A100 (80GB) | **再跳一档，成本约为单卡 A100 的 4 倍以上** |

这里有个重要发现：**14B (INT8) 是一个成本甜点**。它可能刚好挤进 A10 (24GB)，性能比 7B 强很多，但成本与 7B 相同。而一旦超过 A10 的显存上限，就必须跳到 A100，月费直接翻 5-6 倍。

反过来说，**如果已经决定用 A100，那么跑 14B 和跑 32B 的成本是一样的**——都是一张 A100 的钱。这时候没有理由选小的。

### 怎么做最终决定：实测

理论分析只能缩小范围，最靠谱的方式是**用真实数据测试**：

1. 准备 20-30 条有代表性的测试用例（简单的、复杂的、边界情况）
2. 用 API 分别调用不同级别的模型（花费不到几块钱）
3. 重点对比**最难任务**的输出质量
4. 如果小模型质量够用 → 省钱选小的；如果明显差 → 选大的

实测能帮你避免两种错误：选太大（每月多花很多钱）或选太小（质量不达标需要返工）。

## GPU 选型决策树

```
要部署什么模型？
│
├── Embedding 模型（≤8B）
│   └── FP16 显存 ≤ 16 GB
│       └── 选 1x A10 (24GB)
│           └── 选最便宜的 CPU/内存配置即可
│
├── LLM ≤ 14B
│   └── FP8 显存 ≤ 14 GB
│       └── 选 1x A10 (24GB)
│
├── LLM 14B-32B
│   └── FP8 显存 16-38 GB
│       └── A10 装不下 → 选 1x A100 80GB
│
├── LLM 32B-70B
│   └── FP8 显存 32-70 GB
│       └── 选 1x A100 80GB（FP8/INT8）
│       └── 或 2x A100 80GB（FP16）
│
└── LLM > 70B（如 235B MoE）
    └── FP8 显存 > 100 GB
        └── 多卡 A100 或 H100 → 成本极高，先考虑是否真的需要
```

## 总结：两步选型法

1. **先算显存**：参数量 x 精度对应的字节数 x 1.2 = 所需显存
2. **再选卡**：找能装下这个显存的最便宜的单卡方案
