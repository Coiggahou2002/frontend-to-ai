# 从预训练到好用：大模型后训练完全指南

2026年5月2日

> SFT、RLHF、DPO 到 GRPO 的原理与实践

---

你可能见过这样的场景：一个拥有千亿参数的预训练语言模型，你问它"帮我写一封请假邮件"，它却回复了一段维基百科式的科普文章，解释什么是请假制度。它拥有惊人的语言能力，却不知道你在跟它对话。

这就是预训练模型的尴尬——它学会了语言的统计规律，却没学会"当一个好助手"。从 GPT-3 到 ChatGPT，模型参数没有变多，但用户体验发生了质变。这中间的关键，就是**后训练（Post-Training）**。

后训练不是锦上添花。它是一套将"原始能力"转化为"可用产品"的系统工程，包含监督微调、偏好优化、推理强化和安全对齐等多个环节。理解这套体系，你就能明白为什么同样的基座模型，不同团队训出来的效果天差地别。

本教程将带你走完后训练的核心路径：从最基础的 SFT，到经典的 RLHF，再到更轻量的 DPO 和前沿的 GRPO，最后落到数据、安全和评估的实践层面。每一章解决一个具体问题，读完后你能为自己的项目选择合适的后训练方案。

## 第1章 后训练全景——一张图看懂从预训练到部署

### 1.1 后训练的定义与边界

后训练是指在预训练（Pre-Training）完成之后，通过额外的训练步骤让模型具备特定能力的所有技术的总称。预训练教会模型"语言是什么"，后训练教会模型"该怎么用语言"。

一个常见的误解是把后训练等同于"微调"。实际上，微调（Fine-Tuning）只是后训练的一个子集。现代后训练是一个多阶段、可组合的技术栈，不同阶段解决不同问题。

### 1.2 四个阶段的模块化栈

当前业界的后训练流程已经形成了一套相对成熟的模块化架构：

| 阶段 | 目标 | 典型方法 | 解决的问题 |
| --- | --- | --- | --- |
| 监督微调（SFT） | 指令遵循 | 指令数据 + 有监督训练 | 模型不会对话、不遵循格式 |
| 偏好优化 | 价值对齐 | RLHF / DPO / KTO | 模型回答质量不稳定、不符合人类偏好 |
| 能力强化 | 推理与规划 | GRPO / RLVR | 模型不会深度思考、逻辑链薄弱 |
| 安全对齐 | 无害与可控 | 红队 + 安全 RLHF + 规则约束 | 模型可能生成有害、虚假或偏见内容 |

这四个阶段并非必须全部执行，也不一定严格按顺序。一个轻量级的聊天助手可能只需要 SFT + DPO；一个推理模型可能需要 SFT + GRPO + 安全对齐。选择哪些模块、怎么组合，取决于你的目标。

### 1.3 为什么后训练决定了模型的最终体验

InstructGPT 的实验给出了一个惊人的数字：经过后训练的 1.3B 参数模型，在人类评估中优于未经后训练的 175B 参数 GPT-3。这说明后训练的边际收益远大于单纯增加预训练规模。

到 2025-2026 年，行业共识已经从"谁的模型更大"转向了"谁的后训练更好"。DeepSeek-R1 用 GRPO 训练出了接近 o1 水平的推理能力，Meta 的 Llama 系列每一代的核心改进都集中在后训练环节。

## 第2章 监督微调（SFT）——教模型说人话

### 2.1 从"补全"到"对话"的关键跳跃

预训练模型的本质是一个文本补全器：给定前文，预测下一个 token。它不知道自己是一个助手，也不理解"用户在提问"这件事。

SFT 的核心思想很简单：用一批"好的对话示例"来训练模型，让它学会对话的模式。这些示例通常是 instruction-input-output 三元组的形式：

```
指令: 请将以下英文翻译成中文
输入: The quick brown fox jumps over the lazy dog.
输出: 敏捷的棕色狐狸跳过了那只懒狗。
```

训练目标和预训练一样是 next-token prediction，但训练数据从无标注的网页文本变成了精心构造的指令-回复对。

### 2.2 SFT 数据的准备

SFT 数据的质量几乎决定了这个阶段的上限。IBM Research 在 NeurIPS 2025 上的实证研究表明，用更少但更高质量的数据（减少 14%），可以达到甚至超过大数据集的性能。

关键的数据质量维度包括：

- **准确性**：回答必须事实正确，避免幻觉
- **多样性**：覆盖多种任务类型（问答、翻译、摘要、代码、推理等）
- **复杂度梯度**：从简单到复杂的混合，避免全是简单任务
- **格式规范**：统一的对话模板，让模型学会结构化输出

### 2.3 训练实践中的经验法则

SFT 训练相比预训练要轻量得多，但仍有几个关键参数：

| 参数 | 常见范围 | 注意事项 |
| --- | --- | --- |
| 数据量 | 1万 - 10万条 | 质量重要性远大于数量 |
| 学习率 | 1e-5 至 5e-5 | 过大导致遗忘，过小收敛慢 |
| 训练轮数 | 1-3 个 epoch | 多轮容易过拟合 |
| Batch size | 根据显存调整 | 梯度累积可模拟大 batch |

### 2.4 SFT 的常见陷阱

**灾难性遗忘**：过度训练会让模型丢失预训练学到的通用能力。InstructGPT 论文中使用 PPO-ptx（混入部分预训练数据）来缓解这个问题。

**格式锁定**：如果训练数据全是某种固定格式，模型会变得只能按这种格式回答，丧失灵活性。

**过拟合到表面模式**：模型可能学会了"看起来像好回答"的模式（比如总是先说"好的，我来帮你"），但实际内容质量没有提升。

> **检查点**：如果你只有 5000 条高质量数据和 50000 条一般质量数据，你会怎么选？为什么？

## 第3章 RLHF——用人类偏好训练模型

### 3.1 SFT 的天花板

SFT 能教会模型对话，但它有一个根本局限：对于同一个问题，可能存在多种不同质量的回答，而 SFT 只能教模型模仿"示例回答"，无法告诉模型"这个回答比那个好"。

RLHF（Reinforcement Learning from Human Feedback）的核心贡献就是引入了"比较"这个维度——不是告诉模型标准答案，而是让人类标注员对多个回答进行排序，然后用强化学习让模型学会生成"被人类偏好的"回答。

### 3.2 InstructGPT 的三阶段方法

2022 年 OpenAI 发表的 InstructGPT 论文奠定了 RLHF 的工业化基础。它的三阶段流程至今仍是理解后训练的最佳起点：

**阶段一：SFT**。用约 13000 条人工编写的高质量指令-回复数据微调 GPT-3。

**阶段二：训练奖励模型（RM）**。让模型对同一个问题生成多个回答，由人类标注员对这些回答排序。用排序数据训练一个奖励模型，它的输出是一个标量分数，表示"这个回答有多好"。

**阶段三：PPO 优化**。把奖励模型当作"环境"，用 PPO（Proximal Policy Optimization）算法让语言模型生成更高奖励的回答。同时用 KL 散度惩罚防止模型偏离太远。

### 3.3 PPO 的核心机制

PPO 是 RLHF 中最常用的强化学习算法。它的核心思想是"小步更新"：每次更新策略时，限制新旧策略之间的差距，避免灾难性的大幅偏移。

PPO-CLIP 的目标函数通过裁剪概率比率来实现这一点：当新策略相对旧策略的变化超过阈值（通常 ε=0.2），梯度会被截断。这让训练过程更加稳定。

### 3.4 RLHF 的工程挑战

RLHF 在理论上很优雅，但工程实践中困难重重：

- **需要同时维护多个模型**：语言模型、奖励模型、参考模型、价值网络，GPU 显存需求巨大
- **训练不稳定**：奖励模型的偏差会被 RL 过程放大，导致"奖励黑客"（reward hacking）——模型找到高奖励但低质量的捷径
- **超参数敏感**：KL 惩罚系数、学习率、采样温度等参数需要仔细调节
- **人工标注成本高**：高质量的偏好排序数据需要经过培训的标注员

这些挑战催生了后续更简单的方法，比如 DPO。

> **检查点**：奖励模型的作用是什么？如果奖励模型本身不准确，会发生什么？

## 第4章 DPO 及其变体——更简单的偏好学习

### 4.1 DPO 的核心直觉

2023 年，Rafailov 等人提出了一个关键洞察：**语言模型本身就是一个隐式的奖励模型**。你不需要单独训练一个奖励模型，然后用 RL 去优化——可以直接用偏好数据训练语言模型。

DPO 的数学推导从 RLHF 的最优策略出发，通过变量替换推导出一个闭式解。最终的训练目标变成了一个简单的分类损失：给定一对"好回答"和"坏回答"，让模型增加好回答的概率、降低坏回答的概率。

### 4.2 RLHF 与 DPO 的流程对比

| 维度 | RLHF (PPO) | DPO |
| --- | --- | --- |
| 训练流程 | SFT → RM → PPO | SFT → DPO |
| 需要的模型 | 4 个（策略、奖励、参考、价值网络） | 2 个（策略、参考） |
| 数据类型 | 排序数据 + 在线采样 | 离线配对偏好数据 |
| 训练稳定性 | 不稳定，需要大量调参 | 相对稳定 |
| 计算成本 | 高 | 中等 |
| 在线探索 | 有（通过采样新回答） | 无（只用静态数据） |

DPO 的最大优势是简化：不需要训练奖励模型，不需要复杂的 RL 循环，用一个标准的训练循环就能完成偏好优化。

### 4.3 DPO 的局限

DPO 并非 RLHF 的完美替代：

- **数据分布敏感**：DPO 使用离线数据，如果训练数据的分布与模型当前策略差距太大，效果会打折
- **过拟合风险**：在小数据集上容易过拟合，特别是当"坏回答"并不是真的很差时
- **缺乏在线探索**：PPO 能在训练过程中采样新回答并获得反馈，DPO 只能从静态数据中学习

### 4.4 DPO 的变体家族

DPO 的简洁框架催生了一系列变体，每个解决特定的痛点：

| 方法 | 核心改进 | 适用场景 |
| --- | --- | --- |
| IPO | 用 MSE 损失替代 sigmoid，增加鲁棒性 | 噪声偏好数据，坏回答不是很差 |
| KTO | 支持非配对数据（只需"好/坏"标签） | 难以获取配对偏好数据的场景 |
| SimPO | 去掉参考模型，用长度归一化对数概率 | 显存有限，追求效率 |
| ORPO | 将 SFT 和偏好优化合并为一步 | 希望减少训练阶段 |

### 4.5 选择建议

一个实用的选择框架：有充足计算资源和在线采样能力时，PPO/GRPO 通常效果更好；资源有限、追求简单时，DPO 或 SimPO 是合理选择；只有非配对数据时，KTO 是唯一选项。

> **检查点**：如果你有一批配对偏好数据但 GPU 资源有限，你会选择 RLHF 还是 DPO？为什么？

## 第5章 GRPO 与推理强化——让模型学会深度思考

### 5.1 从 DeepSeek-R1 说起

2025 年初，DeepSeek 发布了 R1 模型，展示了一个令人震撼的发现：通过纯 RL 训练（不需要人工编写的推理示例），模型可以自发涌现出链式思考（Chain-of-Thought）能力。这背后的核心算法就是 GRPO。

### 5.2 GRPO 的核心创新

GRPO（Group Relative Policy Optimization）最初在 DeepSeekMath 论文中提出，它对 PPO 做了一个关键简化：**去掉价值网络（critic），用组内相对比较来估计优势函数**。

具体来说，对于每个输入问题，GRPO 会让模型生成一组回答（通常 64 个），然后用奖励函数（或验证器）给每个回答打分。每个回答的优势值不是绝对的，而是相对于这一组的平均水平：

```
优势_i = (奖励_i - 组内均值) / 组内标准差
```

这个设计有两个好处：一是完全消除了价值网络，节省了大量显存；二是组内归一化提供了自然的基线，减少了梯度方差。

### 5.3 RLVR：用验证器替代人类

GRPO 最强大的应用场景是 **RLVR（Reinforcement Learning with Verifiable Rewards）**。在数学和代码任务中，你不需要人类来判断回答好不好——单元测试、证明检查器或数学验证器可以提供完全客观的奖励信号。

这使得训练可以完全在线进行：模型生成回答 → 验证器打分 → 更新策略 → 生成新回答，循环往复。DeepSeek-R1 正是通过这种方式，在数学和代码推理任务上实现了快速提升。

### 5.4 DAPO：解决推理训练的不稳定性

在大规模推理训练中，GRPO 仍然面临稳定性问题。ByteDance 提出的 DAPO 通过四个技术来解决：

1. **防止熵坍缩**：避免模型输出变得过于确定，丧失探索能力
2. **动态批次过滤**：丢弃无信息的训练样本
3. **token 级梯度计算**：比序列级更精细的梯度信号
4. **长度奖励调整**：避免模型学会"写更长的回答来获得更高奖励"

### 5.5 推理强化的边界

RLVR 的效果在有验证器的任务上非常显著，但它有明确的适用边界：

- **适合**：数学推理、代码生成、形式化证明、结构化输出
- **不太适合**：开放式写作、创意任务、主观问答
- **正在探索**：多步规划、工具使用、代理行为

> **检查点**：GRPO 和 PPO 的最大区别是什么？RLVR 为什么特别适合数学和代码任务？

## 第6章 数据工程——后训练的胜负手

### 6.1 质量比数量更重要

这可能是后训练中最反直觉的发现之一：**更少但更好的数据，往往优于更多但质量参差的数据**。

IBM Research 对两个主流开源后训练数据集（Tulu-3-SFT-Mix 和 SmolTalk）进行了详细质量审计，设计了一套数据混合配方（TuluTalk），用比任一原始数据集少 14% 的样本，在关键基准上达到或超过了它们的性能。

### 6.2 SFT 数据的质量维度

构建高质量 SFT 数据集需要关注以下维度：

| 维度 | 说明 | 常见问题 |
| --- | --- | --- |
| 准确性 | 事实正确，无幻觉 | 使用 GPT 生成的数据可能包含错误 |
| 多样性 | 覆盖多种任务和领域 | 数据集偏向某类任务 |
| 复杂度 | 包含不同难度级别 | 全是简单问答，缺乏深度推理 |
| 回复质量 | 详细、有结构、有帮助 | 回复过短或缺乏解释 |
| 安全性 | 不包含有害内容 | 未经过滤的网络数据 |

### 6.3 偏好数据的收集

偏好数据（用于 RLHF/DPO）有两种主要来源：

**人工标注**：请标注员对模型的多个回答进行排序。优点是质量可控，缺点是成本高、速度慢。InstructGPT 的标注团队需要经过专门培训。

**合成生成**：用更强的模型（如 GPT-4）来判断回答质量，或用模型自身生成正负样本。HuggingFace 的 Distilabel 等工具支持大规模合成数据生成。

### 6.4 合成数据的方法与风险

合成数据正在成为主流，但需要警惕几个风险：

- **模型坍缩**：如果模型只从自己生成的数据中学习，可能逐渐丧失多样性
- **幻觉放大**：合成数据中的错误会在训练中被强化
- **同质化**：所有模型都从同一个强模型蒸馏，导致生态多样性下降

SPIN（Self-Play Improvement）通过让模型区分自己的输出和人类输出来自我改进。SPICE 通过在自博弈中引入外部文档来缓解幻觉问题。

### 6.5 数据配方的实践建议

一个可参考的数据混合策略：

1. **基础层**：高质量人工数据，覆盖核心任务类型（占 30-40%）
2. **扩展层**：合成数据，增加多样性和规模（占 40-50%）
3. **校准层**：针对弱点的专项数据（占 10-20%）
4. **去重与过滤**：移除重复、低质量和有害样本

> **检查点**：为什么"用更少但更好的数据"能超过"用更多但一般的数据"？试着用过拟合和噪声的角度解释。

## 第7章 对齐与安全——让模型既有用又安全

### 7.1 安全与有用的张力

后训练中最困难的问题之一，是在"安全"和"有用"之间找到平衡。

一端是过度拒绝：模型对任何敏感话题都说"我无法回答这个问题"，即使问题是合理的医学咨询或安全研究讨论。另一端是过度顺从：模型对有害请求照单全收，成为生成恶意内容的工具。

好的对齐不是简单地让模型学会拒绝，而是让它能够**判断意图、评估风险、选择合适的回应方式**。

### 7.2 安全对齐的主要方法

| 方法 | 原理 | 优势 | 局限 |
| --- | --- | --- | --- |
| 安全 SFT | 用安全对话示例微调 | 简单直接 | 容易过度拒绝 |
| 安全 RLHF | 在偏好中加入安全维度 | 灵活，可多目标优化 | 标注成本高 |
| Constitutional AI | 用规则让模型自我审查和修正 | 可扩展，减少人工标注 | 规则设计困难 |
| 红队测试 | 主动寻找模型的安全漏洞 | 发现真实风险 | 只能发现已知攻击模式 |
| MOSAIC 框架 | "计划-检查-行动或拒绝"流程 | 结构化决策，适合代理场景 | 增加推理开销 |

### 7.3 浅层安全对齐的脆弱性

近期研究揭示了一个令人不安的事实：许多安全对齐技术只影响模型输出的前几个 token，而在更深层的生成中，模型仍可能偏向不安全的方向。这就是为什么"越狱"攻击（jailbreak）能够绕过安全机制——它们本质上是诱导模型跳过那些学到了"拒绝"的初始 token。

DPO 在安全对齐中也有局限：其损失函数对"学会拒绝"这个任务并不是最优的，因为它优化的是配对比较，而不是绝对安全标准。

### 7.4 多目标优化的实践

现代安全对齐需要同时追求多个目标：有用（helpful）、无害（harmless）、诚实（honest）。实践中通常采用分层策略：

1. **基础层**：通过 SFT 建立基本的安全行为模式
2. **优化层**：通过多目标 RLHF/DPO 平衡有用性和安全性
3. **防御层**：通过红队测试、输入过滤和输出审查建立运行时防护
4. **监控层**：部署后持续监控异常模式，迭代改进

> **检查点**：为什么"越狱"攻击能成功？这对安全对齐方法的选择有什么启示？

## 第8章 评估与选型——判断后训练效果的方法论

### 8.1 为什么评估后训练效果很难

评估一个后训练过的模型，比评估预训练模型难得多。预训练可以看困惑度（perplexity），但后训练的目标是多维的——要有用、安全、准确、自然、符合格式，这些维度很难用一个数字概括。

### 8.2 评估方法的三个层次

**自动指标**：快速但粗糙。困惑度、BLEU、ROUGE 等传统指标只能衡量表面匹配，无法评估回答是否真正有帮助。它们更适合作为快速筛选工具，而不是最终判断标准。

**任务基准**：结构化但有限。MT-Bench 和 Arena-Hard 使用多轮对话评估不同能力维度；MMLU 测试知识广度。2026 年业界有约 15 个主流基准在活跃使用，但只有 4 个能可靠预测生产表现。

**人类评估和对战平台**：最接近真实但成本最高。Chatbot Arena 通过 600 万+用户投票的盲评对战来计算 Elo 评分，是目前最接近"真实用户偏好"的评估方式。

### 8.3 评估的实践建议

| 评估场景 | 推荐方法 | 注意事项 |
| --- | --- | --- |
| 快速迭代中 | 自动指标 + 小规模人工抽检 | 不要只看数字 |
| 版本发布前 | MT-Bench + 领域专项测试 | 覆盖安全和边界情况 |
| 方法选型 | Arena-Hard + 人类盲评 | 确保评估集与目标场景匹配 |
| 安全审计 | 红队测试 + 已知攻击模式库 | 持续更新攻击向量 |

### 8.4 后训练方法的选型决策

最后，把所有方法放在一起做一个实用的选型参考：

| 你的情况 | 推荐方法 | 理由 |
| --- | --- | --- |
| 首次后训练，资源有限 | SFT + DPO | 流程简单，工具成熟 |
| 追求对话质量，有标注预算 | SFT + RLHF (PPO) | 在线学习效果更好 |
| 提升推理能力，任务可验证 | SFT + GRPO/RLVR | DeepSeek-R1 验证了有效性 |
| 只有二元反馈数据 | SFT + KTO | 无需配对偏好数据 |
| 显存紧张，快速实验 | SFT + SimPO | 不需要参考模型 |
| 生产级全方位优化 | SFT + RLHF/GRPO + 安全对齐 | 多阶段组合达到最佳效果 |

> **检查点**：如果你要为一个客服场景的大模型做后训练，你会选择哪种方法组合？为什么？

## 实践篇 动手试试后训练

### 用 TRL 库完成一次 SFT 微调

使用 HuggingFace 的 TRL（Transformer Reinforcement Learning）库，在一个小模型上实践 SFT：

1. 选择一个小模型（如 Qwen2.5-1.5B 或 Llama-3.2-1B）
2. 准备 1000 条高质量指令-回复数据
3. 使用 `SFTTrainer` 训练 1-2 个 epoch
4. 对比训练前后的回答质量

关注点：训练损失曲线、回答格式、灾难性遗忘的迹象。

### 用 DPO 对齐一个 SFT 模型

在 SFT 模型的基础上进行 DPO 训练：

1. 准备 500 条配对偏好数据（chosen/rejected 对）
2. 使用 `DPOTrainer` 训练
3. 对比 SFT-only 和 SFT+DPO 的回答质量
4. 尝试调整 β 参数，观察对结果的影响

关注点：回答质量的变化方向、过拟合迹象、β 对结果的影响。
