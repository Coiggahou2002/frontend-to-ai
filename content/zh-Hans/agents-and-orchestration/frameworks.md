# 7. 框架生态

2026 年，agent 框架的故事仍然在变。新库每月都冒出来；老库在不停重组自己的抽象；3 月份的"正确选择"到 9 月可能就掉到中段位。这一页刻意写得短，并且会比本章其他部分更快过时——把它当成一份方位图，不是推荐。

你应该带走的观点是：**在 `anthropic` 或 `openai` SDK 之上自己撸一个薄循环，是完全合理的生产路径。** [§1](./the-agent-loop) 的 agent 循环 ~100 行。加上 [§6](./safety-budgets) 的安全机制，大概 250 行。这个面积小到你能搞懂每一行、能追每一个 bug、能在不重写脚手架的前提下换模型。框架在编排、可观测性和预设模式上有价值；它们的代价是 lock-in、学习曲线和抽象债。要刻意去选。

## 三个具名框架

刻意只挑三个。再多就要做一张这一页撑不到老的对比表。

**LangChain / LangGraph.** 生态分量最重——你能想到的每一个 integration（每个向量库、每个模型 provider、每个工具）。LangGraph 是叠在上面的 agent-graph 层，把 agent 建模成节点之间传状态的有向图。如果你想要"自带电池"、能忍受比较有主张的抽象，且更看重 integration 广度而非极简，就选它。锋利的边角：小版本之间的抽象会动；文档落后于代码；"这是 LangChain 还是 LangGraph 还是 LCEL"的问题。

**OpenAI Agents SDK.** 第一方、轻量、有主张。围绕 OpenAI 的模型和模式（agent 之间的 handoff、内置 tracing、紧凑的循环语义）设计。如果你只用 OpenAI 且想在一套精心设计、现代的 API 上用最少 boilerplate 干活，就选它。锋利的边角：lock-in 是真实的（handoff 模式和 trace 是 OpenAI 特有的）；如果你某个 workload 要换 Claude 或 Gemini，它就不合适。

**Anthropic Claude SDK + 自己的循环。** 这不是框架——只是 SDK 加上本章描述的那 ~250 行胶水。如果你要最大的控制权、最少的 lock-in、并且要能用一行配置改动就换模型（Anthropic / OpenAI / Gemini 的 tool use 在 SDK 形状上足够接近，包一层抽象只要几百行，不需要框架依赖），就选它。锋利的边角：可观测性、重试逻辑，全都你自己写。这对一些团队是 feature，对另一些是 bug。

还有 CrewAI、AutoGen、LlamaIndex 的 agent 模块、Pydantic AI、过去 12 个月里出来的好几个库。它们做的事大致都是本章描述的内容。多数团队在最终敲定之前不会试超过两个。

## 你不需要框架，如果……

- 你的工具不到 5 个、控制模式只有一两种。
- 你能在一个下午写完循环（你能——就在 [§1](./the-agent-loop)）。
- 你不需要花哨的 graph 编排（状态机、并行分支、自带退避的重试）。
- 你宁愿拥有这些抽象，也不愿租用它们。

## 你可能想要框架，如果……

- 你在做带分支、重试、人审检查点的长程规划 agent。
- 你想要预制的 RAG 和 memory 组件，不想自己 integrate。
- 你团队管线上的带宽是零——只有一个工程师兼职做这部分，第三天就需要一个能跑的 tracing UI。
- 你打算长期绑定一家模型 provider，且想要他们第一方 SDK 的体验。

## 真正决定你的 agent 好不好的东西

不是框架。大致按重要性排序是：

1. 工具设计（[§2](./tool-design)）——名字、描述、schema、错误返回。
2. Prompt 质量——系统提示词、工具描述、用户目标怎么 framing。
3. 评估的严格度（[§8](./evaluating-agents)）——有标注集、每次改动都跑、看轨迹。
4. 安全 + 可观测性（[§6](./safety-budgets)）——那些防止你静默回归的乏味事。
5. 模型本身在这个任务上是不是有能力。

框架大概排到第七。挑哪个都行——包括"不挑"——只要让你能把最多时间花在第 1–4 项上。
