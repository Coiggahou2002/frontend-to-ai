# 8. 结语

这是本书的终点。

十二章之前，前提是这样的：一个前端开发者想成为 AI 工程师，而标准建议（"读这些论文、上这门课"）对一个已经会交付软件的人来说形状不对。这本书走了另一条路。从最底下开始——一个 LLM 在机制上到底在做什么——一层一层往上走，走到能在生产环境里构建、提供服务、改动、验证 AI 驱动系统为止。

## 这本书覆盖了什么

- **第 0 章**——LLM 实际上是怎么工作的。Token、下一个 token 预测、对话模板的小把戏、无状态、上下文窗口、采样、非确定性。后面所有内容的机械基础。
- **第 1 章**——给 TS 开发者的 Python。AI 生态的语言，对一个已经会编程的人定制大小。
- **第 2 章**——怎么在代码里调用模型。API、prompt、结构化输出、工具调用、流式、成本，以及生产里会撞上的三种失败模式。
- **第 3 章**——怎么把它没有的知识喂给模型。embedding、向量检索、端到端的 RAG、检索失败模式以及怎么衡量。
- **第 4 章**——怎么让模型有行动能力。工具设计、agent 循环、规划与控制、并行、安全预算，以及评估这一切的"基于轨迹"的方式。
- **第 5–8 章**——硬件和服务的现实。GPU 选型、开源基础设施 stack、KV cache 与前缀缓存、推理并发。
- **第 9–10 章**——改变模型本身。什么时候微调、后训练是什么、LoRA / SFT / RLHF stack 怎么拼起来。
- **第 11 章**——怎么知道这一切到底有没有在 work。让你能安全地改任何东西的纪律。

整张地图就这些：模型在做什么、怎么调它、怎么用知识和工具扩展它、怎么在真实硬件上跑它、怎么改它、怎么衡量它。这条弧足够把你从"我用过 ChatGPT"带到"我能在生产环境里上线 AI 功能"。剩下的是你最后用到哪一块就在哪一块上做深度。

## 书里没有的——往哪里继续

真正的领域比任何一本书都大。一些刻意没覆盖的方向：

- **多模态。**图像、视频、音频的输入和输出。视觉-语言模型、语音模型、生成式图像和视频。机制层面建立在第 0 章的基础上，但实操打法不一样——基于图像的 RAG、音频流式 UX、语音的延迟预算。
- **端侧和边缘推理。**在手机、笔记本、浏览器（WebGPU、MLX、llama.cpp）上跑量化模型。成本结构不同、部署故事不同，基本面相同。
- **从零训练 / 预训练。**这本书一直停在应用和后训练这一侧。完整的预训练 stack——分布式训练、数据流水线、千 GPU 规模的基础设施——是一个独立的学科。
- **特定垂直。**代码 agent、语音 agent、法律文档 AI、医疗 AI、科学发现。每个都有自己一套深度打法，建立在这本书讲的基础之上。
- **基于交互的强化学习。**超出第 10 章里那种后训练风格的 RLHF——完整的 RL 循环，带环境、奖励、自我改进。这是前沿模型工作大量发生的地方，但它在这本书所教的应用工程下面一层。
- **安全、对齐、可解释性。**一个真正的研究方向。第 11 章的评估纪律是从业侧的重叠点。

打好这里的基础之后，你可以挑上面任何一个作为下一个学习项目。

## 如果只能记一件事

如果整本书十二章只能带走一件事，是这个：

> **第三天就把评估集建出来。不是第三周、不是第三个月。**

不能是第三周，因为第三周你已经有用户、已经有"不去测可能不好的东西"的利益。不能是第三个月，因为第三个月你已经依恋你的 prompt 和模型选择，会找各种理由不去测它们。

第三天。趁系统还小、改动还便宜、你还能负担得起在交付压力把答案钉死之前去搞清"好"长什么样。第三天就有 30 个标注 case 的团队，会比第九十天还有 0 个的团队交付更好的产品。

其他东西——模型选哪个、框架选哪个、基础设施选哪个——都有不错的默认。评估纪律没有默认。你只能刻意去建它，而正确的开始时间是在你还没有任何东西可以评估之前。

## 两个实操的下一步

挑一个小项目。不是创业、不是公司、不是 Big Idea。挑**一件有用的事**，让 LLM 来帮，把它交付给十个人。也许是你自己笔记之上的私人 RAG。也许是一个把你日常工作里某项杂活自动化的小 agent。也许是给你团队 repo 的 code reviewer。重点是那个循环：挑一件事、做出来、上线、量它、从真实用户身上学、迭代。一次端到端的循环教你的东西比再读十章都多。

读 `vllm` 的源码，或者 [§7](./tools) 提到的任意一个评估工具的源码。代码比你以为的更好读——它不是魔法，是一群既写 Python 也写 CUDA 的人在做的细致系统工程。读 vllm 里的 paged-attention 实现、或者 Langfuse 的 trace-collection 代码，会让你比再看一份教程更明白这些东西到底怎么 work。在每一个工程学科里都适用的同一条建议在这里也适用：定期在你的领域里读好代码。

## 最后一点反思

这是一个还在动的领域。你 2027 年会用的库可能不是这本书 2026 年点的名。模型会变得更大、更小、更快、更便宜，在某些任务上更好、在另一些上更差，方式无人能预测。一整类目前还不存在的工具会出现。今天存在的某些类别会过时。

第 0 章的基本面——模型机制上在做什么——不会变。第 11 章的纪律——衡量的是分布、不是值；是比率、不是布尔——也不会变。中间那条弧需要随着库和供应商的更替定期重新学一遍。

这其实是好消息。如果你已经把基础和评估纪律内化了，剩下的是你可以跟得上的旋转表面。新模型？跑 golden set。新框架？trace 它。新部署目标？算它的成本。技能会复利。

欢迎来到这个领域。
