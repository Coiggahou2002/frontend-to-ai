# 1. 评估的心态

心态切换一句话就能讲完：

> **是分布，不是值。是比率，不是布尔。**

把这句话内化掉，本章剩下的内容就是机械操作。跳过它，你会花一个季度去写那些还没合并就已经过期的快照测试。

## 论点，重申一次

来自 [第 0 章 §6](../how-llms-work/sampling)：模型输出的是下一个 token 的概率分布。采样从中挑一个。同样的 prompt，调用两次，输出不一样。即便温度 0，批量矩阵乘法里浮点数的非结合性、推理服务器的 batching 方式，也会让结果飘。

LLM 调用的输出不是一个值。它是从一个分布里抽出来的一个样本。所以"它有没有 work？"这个问题，根本不是"输出有没有等于期望字符串？"——而是"从这个分布里抽出来的样本，有多少比例满足我关心的标准？"

这是另一个问题。它需要另一套工具。

## 单元测试思路还能用的地方

不是所有输出都是生成式的。很多由 LLM 驱动的代码路径仍然有"对的答案"：

- **分类。**"这封邮件是不是垃圾邮件？"答案在 {spam, not spam} 里。Accuracy、F1。
- **抽取。**"从这张发票里把公司名称和金额拿出来。"逐字段相等。JSON schema 校验。
- **工具调用的参数抽取。**"agent 是不是用 `amount=100` 调了 `transfer_money`？"对工具日志做程序化检查（[第 4 章 §8](../agents-and-orchestration/evaluating-agents)）。
- **意图路由。**"这条用户消息属于这 12 个意图里哪一个？"混淆矩阵。

对这些，传统 QA 的纪律仍然适用。建一组 `(input, expected_output)` 对，跑模型，数 exact match。非确定性是真的存在，但答案空间小到你能够把它钉死。

把它们当成 LLM 评估里**容易的那一半**。只要适用，永远先做它们——便宜、快、没有歧义。

## 单元测试思路失效的地方

难的那一半，是所有生成式的东西：

- 一段摘要。任何文档都有上百种好的摘要写法。
- 一条聊天回复。模型用一千种措辞都能"有帮助"。
- 一条 RAG 答案。事实必须是对的，但围绕事实的散文是开放的。
- 一条 agent 轨迹。能到达同一目标的工具调用序列有很多种。
- 一项开放式写作任务。根本没有期望输出。

对这些，相等断言是没用的。模型可以是正确的，同时给出十种不同的正确输出。它也可以错得看起来很正确。你需要的是：

- 一份 **rubric**——"好"是什么意思，拆解成可检查的若干维度？
- 一个 **grader**——模型或者人，按 rubric 打分。
- 一种 **分布视角**——把一百个输出都打分，看通过率，而不是看一次跑出来的结果。

这一块就是大部分工作。本章余下的内容讲的就是怎么做这件事而不让自己疯掉。

## 和科学方法的类比

把每一次 prompt 改动都当成一个假设。把这个类比明确写出来：

| 软件工程              | LLM 工程                              |
|---|---|
| 代码改动              | Prompt / 模型 / 工具 / 微调改动        |
| 单元测试              | 标注集合上的程序化 metric              |
| 集成测试              | 标注集合上的 LLM-judge metric         |
| Code review           | regression 指标对 baseline 的 diff     |
| Feature flag 灰度     | shadow traffic、canary、A/B test       |
| 生产监控              | tracing、结构化日志、漂移检测           |

每一次 prompt 改动都是一次实验。标注集合是你的对照组。指标是你的测量。对上一次绿色构建的 diff 就是结果。

如果你说不出假设是什么（"这次改写会把'医疗问题'切片上的幻觉率从 8% 降到 4% 以下"），你就不是在做实验。你只是在凭感觉乱拧旋钮。

## 有评估能避免的两种失败模式

没有评估纪律时，会发生两件事，而且都是悄无声息的。

**沉默的回归。**有人——可能就是你——为修一个客户报告的 bug 改了系统提示词。bug 修好了。同时另外三个没人盯着的切片上的旧行为也消失了。两周后，客服开始收到工单。这时你得在模型版本、prompt 版本、工具 schema 版本之间做二分查找——而这些都没被记录下来。你会花一周时间去恢复。

**Cargo-cult 改动。**一篇博客说"加一句 'think step by step' 能提性能"。你加了。又有人说"Anthropic 推荐在系统提示词里用 XML tag"。你也加了。又有人说"用 0.3 温度，别用 0"。你改了。现在你的 prompt 比该有的长 40%，你说不清哪一项改动起了作用，延迟还涨了。你改了系统五次，什么都没学到。

这两种失败模式根因一样：没有测量。评估就是这两种问题的解药。

## 有评估你能做什么

| 有评估                                              | 没评估                                       |
|---|---|
| 自信地交付一次 prompt 改动                          | 怀着希望交付一次 prompt 改动                 |
| 为了省钱从 Sonnet 换到 Haiku                        | 不敢碰那个模型选择器                         |
| 为了 prompt cache 稳定性重构系统提示词（[第 7 章](../kv-cache)） | 因为它"work"了所以不敢碰                     |
| 在同一份标注集合上对比两个 RAG 检索器              | 互相吵哪个"感觉更好"                         |
| 跑一次微调，知道它到底有没有帮助（[第 9 章](../fine-tuning)）   | 调一周、上线、生产环境里发现问题             |
| 检测到模型供应商一次悄无声息的更新搞坏了你           | 11 天后从客服工单里发现                      |
| 一个 commit 回滚一次回归                            | 横跨三个系统二分一周                         |

直白点说：**没有评估，你就没法安全地改这个系统**。有评估之后，每个组件——prompt、模型、工具、检索器、微调——都变成了一个有可量化行为的可替换零件。

## 心态，再说一遍

呼应 [第 0 章 §6](../how-llms-work/sampling) 和 [第 2 章 §9](../llm-apis-and-prompts/failure-modes)：

```
Hallucination rate.        Not "does it hallucinate".
Injection success rate.    Not "is it secure".
Refusal false-positive %.  Not "did this prompt refuse".
Faithfulness rate.         Not "is the answer right".
Task success rate.         Not "did the agent work".
```

一切关于 LLM 系统的问题，本质上都是关于"在某个输入种群上的某个比率"的问题。把种群建出来（golden set，[§2](./golden-sets)）。把比率测出来（指标，[§3](./metrics)）。让比率在时间上能被追踪。这就是评估这件事的全部工作。
