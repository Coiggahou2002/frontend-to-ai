# 2. 选择提供商

2026 年，一个第一次做 AI 功能的前端开发者，真正要在两类选择中挑：

- **闭源模型 API** —— Anthropic（Claude）、OpenAI（GPT）、Google（Gemini），以及另外几家。你发请求，拿响应，按每百万 token 付费。一切都帮你跑好了。
- **开源权重，自部署** —— Llama、Qwen、DeepSeek、Mistral。你下载权重，跑在自己的（或租来的）GPU 上，通常用 vLLM 或 SGLang 起服务。扩缩容、batching、KV cache，全套都你自己来。这一栈在第 5–8 章讲。

还有第三类——开源权重，由托管推理提供商（Together、Fireworks、Groq、AWS Bedrock、Cerebras）来部署。从机制上看像一个闭源 API，但底下其实是一个开源模型。在同等质量下，定价通常比前沿闭源模型便宜。

## 决策表

近似值，截至 2026 年的现状。成本数字按每百万 token 美元计，每家取一个中端档位的模型。

## 各自适合什么场景

对前端开发者的第一个 AI 功能来说：**几乎一定是用闭源 API**。一次闭源 API 调用的边际成本（每轮 $0.001–$0.05）和你自起一套推理服务的工程成本相比，可以忽略不计。同时你立刻就能用上最强的模型，自带较强的默认安全对齐。

只有满足下列条件之一时，才考虑自部署：

- **隐私 / 监管** —— 你的数据物理上不能离开你的环境。
- **规模** —— 你每天服务上百万次请求，API 账单已经大到 GPU 的 CapEx + OpEx 反而更划算。
- **定制需求** —— 你需要在闭源模型表现弱的领域做微调（第 11 章），或者用到只有开源模型才支持的特性（自定义解码、结构化生成后端、配合量化 KV cache 的超长上下文）。
- **延迟** —— 闭源 API 因为地理距离和排队，TTFT 有一个硬下限；本地 GPU 上自部署能做到个位数毫秒级 TTFT。

本章我们大体上以闭源 API 的代码为例。自部署的等价写法看起来一样——vLLM 和 SGLang 都暴露一个与 OpenAI 兼容的 HTTP 端点。
