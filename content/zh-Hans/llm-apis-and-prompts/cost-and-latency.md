# 8. 成本与延迟基础

你写的每一行 LLM 代码都对应着每次调用的实打实美元成本。从第一天起就养成估算它的习惯。

## 输入 vs 输出：为什么输出更贵

多数提供商对输出 token 的定价比输入 token 贵 3–5 倍。原因是机制层面的，不是商业层面的：

- **输入 token 走 prefill 阶段。** 全部 N 个输入 token 在一次并行的前向传播中处理完。GPU 拿到一长条矩阵，啃得很快。
- **输出 token 来自 decode 阶段。** 每一个输出 token 都要经过自己一次完整的前向传播。生成 500 个 token，就是 500 次顺序的前向传播。同一条请求里这些 token 没法并行，因为每个 token 都依赖前一个（自回归——第 0 章 §2）。

所以输出不只是"同样的工作，收费更高"。它本身就是每 token 更多的算力。KV cache（第 9 章）是让 decode 还能跑得动的关键——没有它，每个新 token 还得对之前所有 token 重新算一遍 attention。有了它，decode 就变成稳稳的鼓点：每个 token 一次 matmul，KV cache 多一项，循环。**第 9 章和第 10 章**会把机制讲透。

**Prompt caching** 在各家以不同形式出现（Anthropic 的显式 cache 标记、OpenAI 的自动前缀匹配、Gemini 的上下文缓存）。它们都能显著降低聊天会话、带稳定 system prompt 的 RAG、以及 few-shot 流水线的成本——任何长前缀被复用的场景都受益。深入机制在**第 9 章的前缀缓存一节**。眼下你只要知道它存在，并且对输入 token 5 折到 9 折的缓存折扣是真实的。

## 速算：一轮 RAG 聊天

一轮典型的 RAG 增强聊天大致长这样：

- 约 500 token 的 system prompt
- 约 6,000 token 的检索上下文（3–6 个 chunk）
- 约 1,000 token 的对话历史（前面几轮）
- 约 500 token 的用户消息
- 约 500 token 的模型输出

合计：约 8,000 输入 + 500 输出。在上面这些提供商上：

| 模型 | 输入成本 | 输出成本 | 单轮合计 |
|---|---:|---:|---:|
| Claude Sonnet 4.6（无缓存） | $0.024 | $0.0075 | $0.032 |
| Claude Sonnet 4.6（90% 缓存） | $0.0024 | $0.0075 | $0.010 |
| GPT-4.1（无缓存） | $0.020 | $0.005 | $0.025 |
| GPT-4.1（50% 缓存） | $0.010 | $0.005 | $0.015 |
| Gemini 2.5 Pro | $0.010 | $0.0025 | $0.013 |
| Gemini 2.5 Flash | $0.0024 | $0.0006 | $0.003 |
| Llama 3.3 70B（Together） | $0.0070 | $0.0004 | $0.0074 |

粗略估算：每月一百万次聊天轮次时，Claude Sonnet 4.6（无缓存）和 Gemini Flash 的差距是 $32K vs $3K。同一个模型上"有缓存 vs 没缓存"的差距是 3 倍。**这些数字会很快被叠加放大。**模型选型和 prompt 缓存的决策主导着 LLM 账单。

## 延迟：TTFT vs 总延迟

要分别盯两个延迟：

- **TTFT（首 token 时间）** —— 从请求发出，到收到响应第一个字节。主要由网络 + 服务端排队 + prefill 阶段（处理全部输入 token）决定。输入越长，TTFT 线性地变慢。闭源 API 在典型负载下大概落在 200–600 ms TTFT。
- **总延迟** —— TTFT 加上输出生成时间。每个 token 的 decode 速度大致恒定（取决于模型和 GPU）。前沿模型在单条请求上典型是 50–150 token/秒。

对 chat UI 来说，TTFT 是用户能感知到的指标。围绕它做优化：更短的 system prompt、更小的检索上下文、prompt 缓存、流式输出。

对一个 batch 流水线（一夜之间总结一万份文档）来说，总延迟主导，TTFT 不重要。

一个有用的经验式：

```
total_latency  ≈  TTFT  +  (output_tokens / decode_tokens_per_second)
```

对一个 8K 输入 / 500 输出的 Sonnet 级调用：TTFT 约 500 ms，decode 约 80 tok/s → 总共约 7 秒。如果你的 chat UI 没法在不显得"坏掉"的前提下渲染 7 秒钟的工作量，你需要的是流式输出，不是更快的模型。
