# LLM 到底是怎么工作的

在用 LLM 构建应用之前，你需要先搞清楚 LLM 实际在做什么——从机制上、从一次 API 调用的层面去理解。前端开发者对 LLM 的大部分困惑（为什么它会忘事？为什么这么贵？为什么同一个问题答案每次都不一样？为什么有上下文长度限制？）都源于缺少这一层基础。

本章就是这层基础。六个核心概念：

1. Token——模型实际读取的内容
2. 下一个 token 预测——LLM 的全部机制
3. 从续写到对话——一个"只会续写"的模型如何聊天
4. 多轮对话——模型是无状态的，是客户端在重放历史
5. 上下文窗口——1M tokens 到底是什么概念
6. 采样——温度、top-p，以及为什么同样的 prompt 每次得到不同的答案

把这六个概念想清楚之后，本指南后续的内容（RAG、Agent、微调、KV cache、GPU 选型）就都变成了"这一层在基本循环之上的什么位置"的问题。本章刻意跳过了 Transformer 架构的内部细节——重要的是**模型在做什么**，而不是它内部的矩阵乘法是怎么排列的。

## 1. Token——模型实际读取的内容

来自终端用户体验的最常见心智模型是错的：模型不读字符，也不读单词。它读的是 **token**。

一个 token 是一段文本——通常几个字符长，有时是一个完整的短单词，有时是一个片段。从文本到 token 的映射由每个模型自带的**分词器（tokenizer）**确定。大多数现代 LLM 的分词器使用 **Byte Pair Encoding（BPE）** 的某种变体：从原始字节开始，反复合并最频繁的字节对，直到形成一个通常在 32K–256K 之间的词表。

一个对 TS 开发者有用的类比：token 就像 parser 中 lexer 输出的 token。你的源代码是 `const x = 42;`，但 parser 看到的不是字符，而是 `[Const, Identifier("x"), Equals, Number(42), Semicolon]`。Token 是系统真正在其上做推理的单元，字符在它的感知层之下。

### 一个具体的例子

来看这个句子：

```
Tokenization is fundamental to understanding LLMs.
```

用一个典型的 BPE 分词器（这里用 GPT-4 的 `cl100k_base`）跑一遍，得到的结果大概是：

```
["Token", "ization", " is", " fundamental", " to", " understanding", " LL", "Ms", "."]
```

九个 token。注意几点：

- `Tokenization` 被拆成了 `Token` + `ization`。常见词根和常见后缀各自成 token。
- 前导空格是 token 的一部分——`" is"` 是一个 token，不是 `" "` + `"is"`。这就是为什么模型对你空格和换行的位置很敏感。
- `LLMs` 变成了 `LL` + `Ms`。不太常见的拼写会被拆碎。
- 句号自己是一个 token。

换一个分词器不同的模型，拆分结果就会不一样。中文、日文、韩文相对于英文，每个字符通常会产生 2–3 倍的 token，因为它们的字符在 BPE 词表中比较稀有。冷门语言和代码混杂的文本就更糟。

### 一些速算经验

| 内容类型                                     | 大致 token 数                            |
|---|---|
| 英文散文                                     | 约 1.3 token/词，约 4 字符/token         |
| 代码（Python/TypeScript）                    | 约 1.5 token/词，空格和操作符各自占 token |
| 中文 / 日文 / 韩文                           | 约 1.5–2 token/字符                       |
| URL、哈希值、base64                          | 几乎 1 token/字符（很丑）                 |

### 为什么这件事很重要

你会在整个 AI 生态里到处看到 "token"：

- **价格**以 token 计价。API 按每百万输入 token 和每百万输出 token 收费，输出通常贵 3–5 倍。
- **上下文窗口大小**以 token 为单位，不是字符或单词。
- **延迟**随 token 数量增长——输入侧（prefill 阶段）和输出侧（decode 阶段）都是。
- **限流**以每分钟 token 数（tokens-per-minute）计。

LLM 工程里几乎每一个有意义的数字都是 token 数。如果你只从这一节学一件实操的事：在你的开发环境里装一个分词器，时不时把你的 prompt 跑一下，建立"这段东西大概有多少 token"的直觉，就像你已经有"这个 JSON 负载大概多少字节"的直觉一样。

```python
# Python: 用 tiktoken（OpenAI 的分词器）数 token
import tiktoken
enc = tiktoken.encoding_for_model("gpt-4o")
tokens = enc.encode("Tokenization is fundamental to understanding LLMs.")
print(len(tokens))   # 9
print(tokens)        # [4421, 2065, 374, 16188, 311, 8830, 445, 43, 82, 13]
```

每个 token 不过是一个整数 ID，是模型词表中的一个索引。模型的输入是一串整数，输出是另一个整数。其余的一切——编码、解码、人类可读的文本——都是发生在边界上的管道工作。

## 2. 下一个 token 预测——LLM 的全部机制

LLM 做的全部事情，一句话就能说清：

> **给定一串 token，模型输出下一个 token 应该是什么的概率分布。**

这就是全部机制。没有别的了。

具体来说，模型是一个纯函数：

```
输入：token ID 列表           例如 [791, 6864, 315, 9822, 374]   （= "The capital of France is"）
输出：概率向量                 词表中每个 token 对应一项（约 10 万–20 万项）
                              例如 " Paris" -> 0.78
                                   " a"     -> 0.04
                                   " the"   -> 0.03
                                   ...（其他 token 拿到更小的概率）
```

输出是**整个词表上的一个概率分布**。每个 token 都有一个分数。模型从来不说"答案是 Paris"，它说的是"鉴于我看到的所有内容，我那 10 万多个候选 token 各自的概率是这样，其中 Paris 的概率最高"。

### 生成 = 循环

生成多个 token 的回复，无非就是把这个单步预测放进一个循环里：

```
input: [The, capital, of, France, is]
  -> 分布 -> 采样 " Paris"
input: [The, capital, of, France, is, Paris]
  -> 分布 -> 采样 "."
input: [The, capital, of, France, is, Paris, .]
  -> 分布 -> 采样 <end-of-text>
完成。
```

每一步：
1. 把当前序列（到目前为止写出的全部内容）喂给模型。
2. 拿到下一个 token 的概率分布。
3. 从分布中挑一个 token（怎么"挑"见第 6 节）。
4. 把它追加到序列上。
5. 重复，直到采样到停止 token，或达到 `max_tokens`。

这个循环叫做**自回归解码（autoregressive decoding）**。每个输出 token 都是把目前为止生成的全部内容再喂给模型一次产生的。

### 这意味着什么

这个机制带来的几个推论，前端开发者一直低估：

**没有计划。** 模型不会"决定要说什么"再写出来。它写一个 token，看一眼自己刚写的，再写下一个。一段长而连贯的回答是每个后续 token 都是前面所有 token 的合理延续的结果——不是某个大纲的产物。

**没有目标。** 模型并不试图变得"有帮助"、"正确"或"让你满意"。它根据学到的模式产生下一个 token。"有帮助性"是模式本身的一种属性，是训练时学到的（第 3 节会提到，第 10 章后训练会再讲）。

**调用之间没有记忆。** 每次前向传播都是独立的。一次调用结束后，模型的"状态"就消失了。第 4 节会回到这一点。

**没有内省。** 模型在任何深层意义上都不知道自己有多确信。概率分布给出了一种"置信度"，但模型并没有特权访问"我是不是真的知道这件事"。无论底层模式是否支持，它都会输出看起来很自信的文本。这就是为什么会有幻觉。

如果你某一刻觉得 LLM 在"思考"，提醒自己：一次前向传播就是一条把输入整数映射到输出概率的矩阵乘法流水线。没有"思考步骤"。看似有思考，是在它训练时学到的模式足够好的时候，单 token 单 token 生成会自然产出看起来像是经过推理的文本。

## 3. 从续写到对话

模型只会做一件事：续写一串 token。那怎么从这件事里得到一个聊天机器人？

### 基础模型只会续写

预训练的原始产物叫做**基础模型（base model）**（或"续写模型"）。它在一大堆互联网文本和书籍上训练过——它的工作就是预测它看到的任何文本里下一个 token 是什么。喂它 `"The quick brown"`，它大概会输出 `" fox"`。

喂一个训练得好的基础模型 `"User: What is the capital of France?\nAssistant:"`，它确实经常会续写出 `" The capital of France is Paris."`——不是因为它理解"你是助手"，而是因为在它训练的语料里，长得像那样的文本后面通常跟着长得像答案的文本。

这本身已经是一个能用的 LLM 了。但基础模型用来聊天不可靠——它也可能续写另一个用户问题，或者跑题，或者写一条 Reddit 评论。它训练目标是模仿**整个**互联网，不是只扮演一个助手。

### Chat 模型在对话格式上做了进一步训练

现代 chat 模型（GPT-4、Claude、Llama-Instruct、Qwen-Chat 等）是在基础模型的基础上，用特定格式的对话进一步训练的——这个过程叫**后训练（post-training）**或**指令微调（instruction tuning）**（第 10 章会深入讲）。

这种格式使用**特殊 token**来标记角色之间的边界。不同模型家族用不同的标记，但本质做的是同一件事。下面是 Qwen 风格的对话模板在底层长什么样：

```
<|im_start|>system
You are a helpful assistant.<|im_end|>
<|im_start|>user
What is the capital of France?<|im_end|>
<|im_start|>assistant
The capital of France is Paris.<|im_end|>
```

Llama 3 用的是另一套标记（`<|start_header_id|>system<|end_header_id|>...<|eot_id|>`），GPT 模型用的是又另一套，但结构都一样：角色标签、正文、结束标记。

当你用 `messages=[{role: "system", ...}, {role: "user", ...}]` 调用 OpenAI 或 Anthropic 的 API 时，SDK 不过是一层薄薄的封装，**把你的结构化消息按模型的对话模板渲染成一大串 token**，发给模型，让模型续写。

### "系统提示词"就是一段普通文本

一旦你看到对话模板，系统提示词所有的神秘感都会消失。它**不是单独的输入通道**，不是元数据，不是某种特殊的指令层。

它就是 prompt 内部的一段文本，前面带了一个 `system` 角色标签，模型在后训练阶段学会了在这段内容上加大权重。

```
<|im_start|>system
You are a senior backend engineer. Be concise.<|im_end|>
<|im_start|>user
Why is my Postgres query slow?<|im_end|>
<|im_start|>assistant
```

模型看到的是这一整团文本，从它停下的地方（紧挨着 `assistant\n` 之后）继续往下写。"system" 标签是它学会去重视的一个信号，但机制上没有任何特殊的事情发生。

带来的几个含义：

- 系统提示词和用户提示词共用同一份上下文预算，都消耗 token。
- 一条足够长的用户消息很容易把一条简短的系统提示词淹没。
- "Prompt injection" 之所以能起作用，是因为一旦用户文本被拼接进同一个字符串，模型就没有任何架构上的特权方式去区分"这部分是运营方的指令"和"这部分是用户在试图覆盖它"。缓解措施是有的，但都是训练时和 prompt 设计上的防御，不是架构上的保证。

### `assistant` 是一个续写提示

关键的小把戏：SDK 把渲染好的 prompt 在 `<|im_start|>assistant\n` 之后就停住——assistant 这一轮已经开始了，但还没有正文。模型一如既往地，要做的事情是续写。所以它一个 token 一个 token 地生成 assistant 消息的正文，直到它产生结束标记 `<|im_end|>`（或它的模板中等价的标记），到那一刻循环停下。

整个"对话"的抽象就是：把会话格式化成一份对话记录，把 assistant 那一轮留空，让模型自动补全。

## 4. 多轮对话——模型是无状态的

这是本章里前端开发者最需要内化的一节。

> **模型在 API 调用之间没有记忆。** 它是一个纯函数：token 进 -> 分布出。两次连续调用之间不共享任何状态。

多轮对话之所以能工作，是因为**客户端**（你的应用、ChatGPT 的 UI、SDK）**每一次请求都把整段会话历史完整地发过去**。

### 第三轮真实的样子

设想一个三轮对话。你以为它是这样：

```
第 1 轮：用户说 A，模型说 B
第 2 轮：用户说 C，模型说 D
第 3 轮：用户说 E，模型说 F
```

第 3 轮真正在网线上走的内容：

```jsonc
// POST /v1/chat/completions  （第 3 轮请求）
{
  "model": "gpt-4o",
  "messages": [
    { "role": "system",    "content": "You are a helpful assistant." },
    { "role": "user",      "content": "A" },   // 第 1 轮 user
    { "role": "assistant", "content": "B" },   // 第 1 轮模型回复，由客户端重放
    { "role": "user",      "content": "C" },   // 第 2 轮 user
    { "role": "assistant", "content": "D" },   // 第 2 轮模型回复，由客户端重放
    { "role": "user",      "content": "E" }    // 第 3 轮 user（唯一"新"的消息）
  ]
}
```

客户端每一轮都在发 6 条消息。服务端的模型完全不知道这是"第 3 轮"——对模型而言，这只是对一长串 token 的一次前向传播，最末尾跟着一个空的 assistant 槽位让它来填。

如果你不再把之前的轮次发过去，模型就完全不知道你在说什么。它不是"在调用之间记得你"。**对话记录就是记忆，由客户端来维护。**

### 为什么这件事很重要

三个直接的后果：

**1. 每一轮都比上一轮更贵。** 输入是单调增长的。一段长对话到第 20 轮时，你的 prompt 可能有 5 万 token——每一个都要计费，每一个都要在每一轮被模型处理。这就是为什么"长聊天"的累计成本会高得出乎意料，哪怕每条用户消息都很短。

**2. 会话历史是你自己的事情。** 服务端不会替你存（少数选择性接入的例外，比如 OpenAI 的 stateful Assistants API 或 Anthropic 的 Memory beta 功能）。如果你的标签页重新加载、本地又没有状态，对话就没了。这就是为什么你只要做一个真正的 chat 应用，就需要数据库——哪怕 LLM API 本身是"无状态又免费"的。

**3. 什么算"历史"由你决定。** 你没有义务必须发完整的对话记录。你可以：
- 截断旧轮次（上下文快满时丢掉最早的几条消息）。
- 总结旧轮次（把 20 条消息替换成一段由模型自己写的 200 token 的摘要）。
- 选择性地检索相关的过往消息（在对话历史上做一种 proto-RAG）。
- 把系统侧信息（工具输出、检索到的文档）作为消息注入进去。

`messages` 数组就是你的草稿板。模型的"记忆"就是你放进去的内容。

### 前向引用

后面有两章直接建立在这一节上：

- **第 2 章（LLM API 与 Prompt 工程）**主要就是在讲怎么把 `messages` 数组设计好——system 角色里放什么、怎么管理历史、怎么注入工具结果、怎么组织 few-shot 示例。
- **第 7 章（KV Cache）和第 8 章（推理并发）**针对的就是"prompt 越来越长"这个成本问题。即使你每一轮都重发整段历史，推理引擎可以识别出共享的前缀，复用对应的 KV cache 中间状态，而不必从头重新计算。这叫做**前缀缓存（prefix caching）**，也是为什么"长 prompt"实际上没听上去那么灾难——但这种优化的存在本身恰恰就是证据：你确实在概念上每次都重发整段对话记录。

## 5. 上下文窗口——1M tokens 到底是什么概念

**上下文窗口**是模型在一次前向传播中能看到的最大 token 数。除了模型大小本身之外，它是 LLM 最具决定性的一个参数。

### 哪些内容算进窗口

请求中的每一个 token 都算：

```
上下文窗口  =  系统提示词
            +  会话历史（之前所有 user + assistant 轮次）
            +  当前用户消息
            +  工具定义（如果有）
            +  检索到的文档 / RAG 片段（如果有）
            +  assistant 正在生成的回复（输出也住在这个窗口里）
```

输出也在窗口里。如果你在一个 200K 上下文的模型上输入了 199K token，那就**只剩 1K token 给响应**——模型会在句子说一半的时候被截断。这就是为什么长上下文模型经常单独有一个比窗口本身小的 `max_output_tokens`（例如 1M 上下文模型可能把输出上限定在 8K 或 32K）。

### 为什么有上限

两个真实的原因，都是硬件层面的：

**1. 计算。注意力是 O(n²)。** Transformer 的注意力机制本质上是让每一个 token 去注意之前的每一个 token。序列长度翻倍，注意力计算量就是 4 倍。从 4K 上下文到 1M 上下文不是 250 倍的工作量，而是 250² = 62,500 倍——这是注意力层的数字（网络其余部分对上下文长度是线性的，但在长上下文下注意力占主导）。推理引擎用 **KV cache**（第 7 章）来缓解这一点，避免在 decode 时对旧 token 重新做注意力计算，但对一段新的长输入做 prefill 时，仍然要付完整的代价。

**2. 显存。KV cache 随上下文长度线性增长。** 推理时模型必须为每个 token 在每一层都存一份 Key 向量和 Value 向量。一个典型的 70B 模型用 BF16 算下来，每个 token 大约 300KB。1M token 上下文就是大约 300GB——已经超过单张 H100 的显存。长上下文不只是"更慢"，它在物理上需要更多的 GPU 显存，这就是为什么长上下文推理要靠多卡部署，加上像 MLA、GQA、FP8 KV cache 这样的激进优化（第 7 章会详细讲）。

这两个限制——平方级的计算和线性的显存——加在一起，决定了"上下文窗口"是一个固定的数，而不是一个可以随便调高的设置。模型架构和可用硬件共同决定了它能多大。

### 1M tokens 到底是什么概念

LLM 的市场宣传里到处甩"1M 上下文"这种数字，却没有任何参照系。对一个英文开发者而言，下面是大致能塞进去多少东西：

| 内容                                                  | 字数        | Token 数（约 1.3 token/词）|
|---|---:|---:|
| 一篇典型的博客文章                                    | 1,500       | ~2,000                 |
| 一篇短篇小说                                          | 5,000       | ~6,500                 |
| 一篇长篇杂志特稿                                      | 10,000      | ~13,000                |
| 一部中篇小说                                          | 30,000      | ~40,000                |
| 《动物农场》（全文）                                  | 30,000      | ~40,000                |
| 一部典型长篇小说                                      | 90,000      | ~120,000               |
| 《了不起的盖茨比》（全文）                            | 50,000      | ~65,000                |
| 《战争与和平》（全文）                                | 580,000     | ~750,000               |
| 哈利·波特全 7 册                                      | 1,100,000   | ~1,500,000             |

对开发者来说，按代码 / 文档来感受：

| 内容                                                  | 大约 token 数   |
|---|---:|
| 一个中等大小的 React 组件文件                         | 500–2,000       |
| 一页内容密集的 PDF                                    | 500–800         |
| 一份 50 页的 PDF 报告                                 | 25,000–40,000   |
| 你这个服务的完整 OpenAPI schema                       | 5,000–30,000    |
| 一个中等规模的 TypeScript 代码库（5 万行）            | 600,000–800,000 |
| 3,000 页内容密集的 PDF                                | ~1,000,000      |
| 一个人一整年的邮件（约 1 万封）                       | ~1,500,000      |

**1M token 大致相当于：整套哈利·波特、或者一个中等代码库、或者大约三千页内容密集的 PDF。** 这是当前前沿模型一次前向传播能容纳的上限。

### 上下文窗口的历史

| 年份      | 模型                       | 上下文窗口         |
|---|---|---:|
| 2020      | GPT-3                       | 2,048              |
| 2022      | GPT-3.5                     | 4,096              |
| 2023      | GPT-4（初版）               | 8,192              |
| 2023      | Claude 2                    | 100,000            |
| 2023      | GPT-4 Turbo                 | 128,000            |
| 2024      | Claude 3                    | 200,000            |
| 2024      | Gemini 1.5 Pro              | 1,000,000–2,000,000|
| 2025–26   | 前沿模型（当前）            | 200K–1M+ 已是常规  |

上下文窗口在 5 年里增长了约 500 倍。这是这个领域提升最快的能力，已经重塑了哪些应用是可行的。

### "长上下文"不等于"全用上"

两条重要的注意事项：

**1. 成本随上下文长度线性增长。** 在一个 1M 上下文模型上，一个 200K token 的 prompt 在输入侧的费用是同一个模型上 1K token prompt 的 200 倍。一次 1M token 的调用，光输入就可能要花 1–3 美元。长上下文是一种能力，不是默认值。

**2. 质量经常随上下文长度下降。** 即使一个模型在技术上支持 1M token，它从那段上下文中间召回和使用信息的能力，往往比从开头或结尾召回要弱——这种现象叫做 **"lost in the middle"**。像 *needle in a haystack* 这样的基准就是直接测试这一点的。有些模型能扛到 200K 后才下滑；有些在检索上很强，但在跨整个窗口做综合时不行。

实际上，"模型真正用得好的 200K 上下文"往往比"中间召回退化的 1M 上下文"更有价值。

### 前向引用

本指南里有三件事直接追溯到上下文窗口的有限性：

- **第 3 章（嵌入向量、向量检索与 RAG）**就是因为这一节而存在的。RAG——检索增强生成——是对"我手上的知识比上下文窗口能装的多"这个问题的架构层答案。与其往 prompt 里塞 1000 万个 token，不如把你的语料嵌入到向量数据库里，再针对当前查询检索出相关的几千个 token。整个 RAG 生态的存在本身就是上下文窗口有限的产物。
- **第 7 章（KV Cache）**解释了长上下文受限的硬件原因——KV cache 大小随序列长度线性增长，GPU 的显存会用完。也解释了让 200K–1M 上下文变得可行的那些优化（GQA、MLA、FP8）。
- **第 8 章（推理并发）**会介绍 `--max-model-len` 这个参数，自部署 LLM 时你用它来决定每个请求最多支持多长的上下文。它直接在"单请求更长"和"并发更多"之间做权衡——两者吃的是同一份 KV cache 显存预算。

## 6. 采样——为什么同样的 prompt 每次得到不同的答案

第 2 节说过，模型输出的是下一个 token 的**概率分布**，它不输出"下一个 token"。所以要真正生成文本，我们需要一步**从分布中挑出一个 token**。这一步叫做**采样（sampling）**，怎么采样会显著影响输出的"性格"。

### 贪心：永远选最高概率的那个

最简单的策略：永远选概率最高的那个 token。

```
分布：{ " Paris": 0.78, " a": 0.04, " the": 0.03, ... }
贪心选：" Paris"
```

这是**确定性的**——同样的输入，每次都得到同样的输出。在你想要可复现性的时候很有用。但贪心解码有个已知的坏毛病：它会产生重复、乏味的文本。因为每一步它都坚持选当下局部概率最高的 token，所以会陷入循环（"the the the the the"），或者写成无聊的样板文。它也很脆弱——如果模型有一点点校准偏差，贪心解码会把这个偏差放大成一条注定走错的路径。

### 温度：重塑分布

模型最原始的输出，在任何 softmax 转成概率之前，是一个未归一化的分数向量，叫做 **logits**。把 logits 变成概率分布要用 softmax。**温度**就是在 softmax 之前对 logits 做缩放的一个数：

```
adjusted_logits = logits / temperature
probabilities   = softmax(adjusted_logits)
```

效果：

- **低温（0.0–0.3）**：用一个小数除 logits，让它们之间的差距变得更大。分布变得**更陡峭**——最高概率的 token 拿到更多概率质量，不太可能的 token 被压得趋近于 0。输出更确定、更保守、更倾向于重复显而易见的答案。T=0 在数学上等价于贪心解码（严格说是它的极限）。
- **T = 1.0**：保留模型原始输出的 logits。这是模型被训练去输出的"默认"分布。
- **高温（1.0–2.0）**：用一个大数除，让分布变平。不太可能的 token 得到了真正被选中的机会。输出更多样、更有创意、更可能让你惊喜——也更可能跑飞。

一个实用的经验法：

| 任务                                              | 推荐温度    |
|---|---|
| 代码生成、结构化抽取、分类、事实问答              | 0.0–0.2     |
| 通用助手任务、技术写作、平衡型对话                | 0.3–0.7     |
| 创意写作、头脑风暴、营销文案                      | 0.7–1.2     |
| 天马行空的点子，"给我 20 个不同角度"              | 1.2–1.5     |

对绝大多数生产用途——RAG、代码生成、结构化抽取、agent 的工具调用——你都希望温度低（0.0–0.3）。创造性是聊天和写作助手的一个 feature，对于需要产出可预测输出的系统来说不是。

### Top-p（核采样）

温度重塑了分布，但仍然把所有 token 都纳入考虑。**Top-p 采样**，也叫**核采样（nucleus sampling）**，思路不同：只考虑累积概率至少为 `p` 的最小 token 集合，把其他 token 全部置零。

```
p = 0.9
分布排序：Paris (0.78), a (0.04), the (0.03), my (0.02), some (0.02), ...
累积：    0.78,         0.82,     0.85,       0.87,      0.89, 0.91 <- 停
核：     { Paris, a, the, my, some } —— 这就是我们采样的集合
```

Top-p 会随分布的形状自适应。当模型很有把握（一个 token 占主导）时，核很小，输出基本上是确定的。当模型不太确定（很多 token 都说得通）时，核会扩大，输出会变得更多样。

实际上，**大多数 API 让你既能设温度也能设 top-p**，但你通常只调其中一个。一种常见起点是 `top_p = 1.0`（关闭）只调温度；或者 `temperature = 1.0` 只调 top-p（常用 0.9）。

还有 `top_k`，思路类似：只考虑概率前 K 高的 token。在现代 API 中没有 top-p 用得多。

### 还会看到的其他参数

| 参数                  | 作用 |
|---|---|
| `frequency_penalty`   | 降低已经出现过的 token 的概率（按出现次数计，抑制重复）。 |
| `presence_penalty`    | 降低任何已经出现过的 token 的概率（鼓励引入新话题）。 |
| `stop`                | 一组字符串列表，一旦输出就停止生成。强制结构化输出格式时有用。 |
| `seed`                | 一些 API 允许你固定采样的随机种子，让 temperature > 0 也可以复现——尽管没有 API 能完美保证这一点。 |
| `logit_bias`          | 手动把特定 token ID 的概率上调或下调。高级玩家功能。 |

### 前端开发者的头疼点：非确定性

从普通软件来到这里最大的转变：**同样的输入，同样的模型，不同的输出**。即使在 temperature 0 下，你也可能拿不到比特级别的可复现——批量矩阵乘法中浮点数的非结合性、kernel 的 autotuning、推理服务器把请求打成 batch 的方式，都会引入小幅波动。

这会打破你习惯了的那套测试方法：

- 快照测试不行——快照会漂移。
- 等值断言不行——同样的意思模型可以用五种不同的说法说出来。
- "跑一次然后检查"不能告诉你是不是出现了回归。

你需要不一样的评估手段：用评分准则给输出打分、把同一个 prompt 跑很多次再看分布层面的指标（正确率、拒绝率、平均输出长度）、用一个独立的"裁判"LLM 来评判输出，并接受你测的是**回归率**而不是相等。

这是 LLM 评估的核心挑战，专门有一章——**第 11 章（评估与可观测性）**。简短版本：别再用单元测试作者的思路想问题，开始用一个有统计意识的 QA 工程师的思路想问题。是分布，不是值。

## 串起来看

一次完整的 LLM API 调用，从头到尾：

```
1. 你的客户端拼出一个 `messages` 数组
   - 系统提示词 + 完整的过往历史 + 新的用户消息
   - 这是你的活；服务端没有之前轮次的记忆

2. SDK 按模型的对话模板把 messages 渲染成一串 token 序列
   - 角色标签会作为特殊 token 插入
   - 序列在 `<|im_start|>assistant\n` 之后立即结束，让模型知道要续写什么

3. 服务端检查 token 数是否超过上下文窗口
   - 太长就给你 4xx 错误（或者 token 被截断，看具体 API）

4. 模型做前向计算
   - "prefill" 阶段：并行处理所有输入 token，产出 KV cache，得到第一个输出 token 的分布
   - "decode" 阶段：采一个 token，追加，再前向，再采下一个……直到停止 token 或达到 max_tokens

5. 每个输出 token 都是采样得到的
   - 温度、top-p 等参数塑造分布
   - 选中的 token 被追加到序列上，再次喂回去

6. 响应以流式（或一次性）返回
   - SDK 剥掉对话模板的包装，把 assistant 文本交还给你

7. 请求一结束服务端就忘了一切
   - 没有任何状态被保留；如果你想要"下一轮"，就再重放一次历史
```

本指南的每一章都建立在这个循环之上：

- **第 1 章（Python）**给你能真正去调用这个循环、处理它的输出、围绕它构建系统的语言。
- **第 2 章（LLM API 与 Prompt 工程）**讲的是怎么"工程化"这个 `messages` 数组——系统提示词放什么、用户消息怎么组织、什么时候用 few-shot 示例、怎么把多次调用串起来。
- **第 3 章（嵌入向量、向量检索与 RAG）**讲的是怎么把对的知识塞进 prompt 里同时又不超出上下文窗口。
- **第 4 章（Agent 与工具调用）**讲的是怎么通过把模型的一部分输出 token 解释成函数调用，让模型与外部世界互动——本质上仍是一次前向传播，但输出被解读成工具调用。
- **第 5–8 章（GPU 选型、基础设施技术栈、KV cache、推理并发）**是这个循环的硬件和服务侧——跑一个模型要花多少钱、KV cache 怎么让服务器跳过重复计算、并发请求之间怎么共享 GPU 显存。
- **第 9–10 章（微调与后训练）**讲的是*改变*模型已经学到的分布——让它在你的特定任务上做得更好，或者用强化学习塑造它的行为。
- **第 11 章（评估）**讲的是在非确定性面前，怎么衡量上面这一切到底有没有起作用。

到这里你理解了这台机器。下一章会教你操作它的语言。
