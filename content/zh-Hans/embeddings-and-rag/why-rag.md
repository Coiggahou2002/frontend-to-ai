# 1. 为什么需要 RAG

[第 2 章](../llm-apis-and-prompts/api-call-shape)里那个模型可以聊天、可以返回 JSON、可以调工具。但它有三个结构性的盲区，光靠 prompt 是绕不过去的：

1. **上下文是有边界的。** 一个 1M token 的模型仍然看不到你 50M token 的语料库（[第 0 章 §5](../how-llms-work/context-window)）。
2. **训练截止日期。** 截止日期之后发生的事情模型一概不知——你上个 sprint 的 PR、今天的故障报告、你团队今早刚发布的 API。
3. **幻觉。** 即使是训练过的话题，模型也会生成听起来合理的文本，而它本身并没有"我是不是真的知道"这种特权信号（[第 2 章 §9](../llm-apis-and-prompts/failure-modes)）。

RAG 用同一个技巧解决这三个问题：在查询时把相关的那一小段文本检索出来塞进 prompt。

## 三个问题，具体来看

### 问题 1：语料 > 上下文窗口

前端开发者的心智模型："上下文窗口是模型的 RAM。" 如果你的知识库比 RAM 还大，你就需要一种"从硬盘取数据"的策略。RAG 就是这个 fetch。

具体一点，假设你公司有 100,000 篇内部文档，平均每篇 2,000 词：

```
100,000 docs × 2,000 words × 1.3 tokens/word ≈ 260,000,000 tokens
```

这是 1M 上下文模型的 260 倍。市面上没有这种模型——大概也永远不会有——能让你每次调用都把这玩意儿全塞进去。就算能，看看成本：

```
260M input tokens × $3 per 1M input tokens = $780 per query
```

你付不起。RAG 把同一个问题变成"embed 一下 query（几分钱）、检索 5 个 500 token 的 chunk、把 2,500 token 的 context 加上 query 发出去"。每次调用几分钱。

### 问题 2：截止日期之后的知识

模型的训练数据有截止日期——通常是发布前 6 到 12 个月。它不知道你公司存在、不知道你代码库长什么样、不知道你昨天修了什么 bug、不知道用户刚在另一个 tab 里输了什么。RAG 是注入时效性知识的标准模式：把最新的文档建索引，让检索按需把它们拉进来。

### 问题 3：幻觉的 grounding

当模型被强制**只能基于你提供的几段 chunk 来回答**时，幻觉率显著下降。它仍然会出错——可能误读、混淆、补全空白——但失败模式现在变成了"忽略或误读了源材料"，这是可以审计的；而不是"凭空捏造"，那是没法审计的。引用模式（[§5](./retrieval-pipeline)、[§8](./production-patterns)）让"可审计"成为整件事的核心目的。

## "为什么不直接把所有东西都塞进 prompt？"

即使语料*确实*能塞下，全塞进去也是浪费：

| 策略 | 每次调用的 token | 延迟 | 质量 |
|---|---:|---|---|
| 把 200K token 的整个代码库塞进去 | 200,000 | 几秒的 prefill | "lost in the middle"——长上下文下召回退化 |
| 检索 top-5 相关 chunk（约 2,500 token） | 2,500 | 几十毫秒 | 更锐利——模型只读真正重要的内容 |

成本是 80 倍。检索通常会让质量*上升*，而不是下降——长上下文模型已知会低估长 prompt 中部的信息。一个简短、精准的上下文胜过一个巨大的 dump。

## "为什么不直接微调？"

一个常见的混淆。微调和 RAG 解决的是不同的问题。

| | 微调 | RAG |
|---|---|---|
| **改的是什么** | 模型权重 | Prompt |
| **最适合** | 行为——语气、格式、风格、领域推理 | 知识——事实、来源、新数据 |
| **更新成本** | 重训（几小时到几天，$$$） | 重新建索引（几分钟，几乎免费） |
| **可审计性** | 难——知识被烤进权重 | 容易——chunk 可以被引用 |
| **幻觉** | 在事实问题上和基模一样 | 在语料中存在的事实上显著降低 |

一条简单的判据：**行为用微调，知识用 RAG**。如果你想让模型永远以法律备忘录格式回答，去微调。如果你想让它回答关于你 50,000 份法律备忘录的问题，去 RAG。大多数生产系统两个都做——一个轻度微调过的模型，消费 RAG 的 context。

微调把知识有损地烤进去——模型"记住"了语料但没法引用，而你也没法在不重训的情况下更新单个事实。微调的端到端流程在**第 9 章**讲。

## 这个名字

这个架构有一篇起源论文——*Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks*（Lewis et al., 2020）——但今天生产环境里所有东西都是同一个想法的变种：**先检索，再阅读**。你会看到它被叫做 RAG、retrieval、grounding、context augmentation、in-context learning with retrieval。同一回事。
